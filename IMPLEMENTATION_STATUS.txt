🎯 WHAT WAS FIXED
────────────────────────────────────────────────────────────────────────────
Problem:   previewEffectiveStake() could return inaccurate voting power if 
           the reputation oracle updated between preview and vote

Root Cause: Both previewEffectiveStake() and vote() query the oracle 
           independently with no mechanism to ensure consistency

Solution:  Added reputation snapshot tracking and optional validation to 
           detect staleness between preview and vote

Impact:    Users can now safely preview voting power and validate that it 
           hasn't changed before voting

🚀 NEW FUNCTIONALITY
────────────────────────────────────────────────────────────────────────────

1. voteWithValidation()
   - Vote with optional reputation staleness validation
   - Configurable drift tolerance (basis points: 0-10000)
   - Timestamp-based staleness check (1 hour default)
   
2. previewEffectiveStakeWithTimestamp()
   - Preview effective stake with block timestamp
   - Enables off-chain staleness tracking
   
3. checkReputationStaleness()
   - Check if reputation has changed or timed out
   - Returns current reputation and time since last vote
   
4. getLastReputationSnapshot()
   - Get last recorded reputation for a user
   - Enables verification of vote reputation
   
5. _validateReputationFreshness() [internal]
   - Core validation logic
   - Checks both drift percentage and timestamp

New Events:
  • ReputationSnapshotRecorded
  • ReputationStalenessValidated
